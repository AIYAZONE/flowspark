# AI 洞察产品化（阻力 Top3 + 行动建议 & 场景页信息架构重排）设计

## Summary
- 目标：把 `/profile/ai-insights` 从“工程可观测性看板”升级为“普通用户可理解、可行动、可反馈并可解释系统变化”的产品页。
- 本次范围：
  - 场景页：默认把“回放 + 反馈”放在用户路径更靠前位置；“技术参考（strategy/model/prompt 对比）”默认折叠，避免干扰。
  - 概览页 + 场景页：新增“主要阻力 Top3 + 对应一条行动建议”模块，帮助用户看完就知道下一步怎么改。
- 约束：不新增依赖；不改接口；不做 DB 迁移；保持现有数据结构兼容。

## Background
- 现状问题：
  - 页面暴露大量内部字段与版本号（strategy/prompt/model/phase_x_v1），普通用户难以理解。
  - 用户看完无法得到明确“下一步怎么做”，因此价值感弱。
  - 虽已有“回放 + 反馈”能力，但入口与叙事不够产品化，用户难以感知“反馈会如何影响系统”。
- 已确认的产品假设：
  - “主要阻力 Top3”以用户反馈标签（feedback_label）为第一数据源，更贴合普通用户理解。
  - Top3 模块需在“概览页 + 场景页”同时提供：全局总结 + 场景聚焦。

## Goals（成功标准）
- 普通用户进入洞察页，能够回答：
  - 我最近主要被什么阻力卡住（Top3）。
  - 我下一步能怎么做（每条阻力对应一条行动建议）。
  - 我反馈会如何影响后续建议（场景页一句话解释）。
- 点击“查看详情”后：
  - 页面直接定位到单条建议回放（含反馈入口）。
  - 技术对比与表格信息默认不抢占注意力（可折叠）。

## Non-goals（不做）
- 不引入新的 AI 生成“行动建议”能力（先规则化建议，保证稳定性与可解释性）。
- 不新增后台定时任务/cron。
- 不改变 recommendation/outcome 的写入结构与现有 API 协议。

## IA（信息架构）

### 场景页 `/profile/ai-insights/[scene]`
展示顺序（从上到下）：
1. 场景标题 + 时间范围切换
2. “系统如何根据你近期反馈做调整”的一句话提示（按该场景 recent 信号生成）
3. 主要阻力 Top3 + 行动建议（新模块）
4. 最近建议列表（可选保留为表格或改为列表形态；本次最小改动优先保留现有表格结构）
5. 单条建议回放 + 反馈（锚点定位）
6. 技术参考（折叠区，默认收起）：strategy/prompt/model 对比表、术语解释

详情模式：
- 当 URL 包含 `rid` 时视为“详情模式”，默认滚动定位到回放区域并优先展示回放；技术参考保持折叠。

### 概览页 `/profile/ai-insights`
展示顺序（从上到下）：
1. 页面说明（用途/怎么用/反馈会影响什么）
2. 全局主要阻力 Top3 + 行动建议（新模块）
3. 场景概览（各场景的采纳/完成等）
4. 最近建议（含“查看详情”）
5. 技术参考（降级展示或折叠）

## Top3 统计口径

### 数据源
- 读取 `getRecentRecommendations()` 的 `feedback_label` 字段。
- 概览页：scene 不过滤；场景页：按当前 scene 过滤。

### 过滤与归一
- 忽略：
  - 空值
  - `useful`
  - `dismiss` / `close_result`
- 归一：
  - `not_fit_goal` / `not_fit_action` / `not_fit_tone` / `not_fit` 统一归一为 `not_fit` 用于 Top3 统计（细分原因仍保留在单条回放的反馈中）。

### Top3 取值
- 统计最近 N 条记录（建议 N=30；按当前 range 或固定窗口两种实现二选一，默认跟随 range）。
- 按频次降序取 Top3。
- 不足 3 项：
  - 只展示已有项；
  - 同时展示“建议多反馈以获得更准确洞察”的提示文案。

## 行动建议（规则化映射）
每个阻力映射 1 条建议文案，保持：
- 一句话
- 可执行
- 不包含技术词

建议映射（示例，可微调语气）：
- `no_time`：
  - zh：下次先选 5/10 分钟版本，只做“开始动作”，先建立连续性。
  - en：Next time choose a 5/10-minute option and do a single “starting move” to build momentum.
- `too_hard`：
  - zh：把建议拆成更小一步，先采纳“最小可执行版”。
  - en：Break it into one smaller step and adopt the “minimum viable” version first.
- `not_fit`：
  - zh：先确认你的目标与现有行动是否更新；反馈里选更具体原因，系统会更贴合你。
  - en：Make sure your goals/actions are up to date; choose a more specific reason in feedback so the system can adapt.
- 其他阻力（未来可扩展）：若出现未知值，显示“请在反馈中选择更具体原因”作为兜底。

## 文案与可解释性
- Top3 模块需要解释一句“这些来自你最近的反馈”，避免用户误解为模型臆测。
- 场景页需要保留一句“系统如何调整”的说明，让用户感知反馈有效。
- 技术参考折叠区标题明确标注“可忽略”，降低心理负担。

## Acceptance Criteria（验收标准）
- 点击“查看详情”进入场景页：
  - 自动定位到回放区域（锚点）。
  - 技术参考默认折叠，不影响阅读回放与反馈。
- 概览页存在全局 Top3 模块，场景页存在场景 Top3 模块：
  - 中文/英文均不出现内部 tag（如 `not_fit_goal`、`phase_b_v1`）作为主信息。
  - 无反馈时有明确空状态引导。
- 不新增依赖；不改接口；生产构建通过（`next build`）。

## Risks & Mitigations
- 用户反馈不足导致 Top3 价值弱：
  - 提供空状态引导；在回放区更显眼位置保留反馈入口。
- 归一 `not_fit_*` 会损失细分洞察：
  - Top3 只做归一统计；单条回放仍保留细分原因用于后续优化与解释。
