# Chat 作为主页面 Design

## 目标

把现有 `/chat` 从“新增的一个对话页”升级为 **认证后默认主页面**，并把视觉从“普通内容页”推进成 **沉浸式主对话场**。

这次设计要同时解决两个问题：

- 结构层：让 `/chat` 成为真正的第一入口，而不是 `/system` 的附属页
- 体验层：让 `/chat` 看起来像主战场，而不是一页里放了一个聊天模块

## 已确认决策

- 认证后默认进入 `/chat`
- 侧边栏第一项改为 `/chat`
- `/chat` 采用“沉浸式对话场”风格
- `/system` 降级为系统总览页，保留信息与入口，不再承担首页角色

## 非目标

- 不改 `/api/command/draft` 的核心能力
- 不引入多会话列表、历史会话管理、搜索、流式输出
- 不扩展新的 agent 编排逻辑
- 不重做现有 `SystemConversationProvider` 记忆模型

## 当前状态

### 路由与入口

- 当前认证后仍是 `/system` 在承担首页心智
- 侧边栏顺序里 `/system` 仍排在 `/chat` 前面
- `/chat` 已存在，但更像一个新页面，而不是主页面

### `/chat` 当前视觉问题

现有 `/chat` 已经实现了“输入和回答在同一页”，但主页面感还不够，主要体现在：

- 顶部仍是明显的内容卡片式介绍块
- 对话流本身没有成为页面绝对中心
- 输入区虽然已经页内固定，但视觉上仍像一个被包在卡片里的功能模块
- 页面整体更像“有聊天功能的内容页”，而不是“系统主战场”

## 设计方向

### 1. 页面定位

`/chat` 要成为认证后用户默认看到的页面，也就是新的“系统入口”。

它的职责应该是：

- 接住自然语言输入
- 呈现系统连续对话
- 成为跨页面回来继续聊的唯一场所

`/system` 的职责则收回为：

- 展示系统总览
- 展示调度与说明信息
- 提供次级入口（例如进入专注模式、进入系统对话）

### 2. 视觉原则

沉浸式主对话场的核心原则：

- 弱化“卡片堆叠感”
- 强化“消息流是页面主轴”
- 强化“输入区是主控制台”
- 保留系统感，但不让头部说明抢走主视觉

换句话说，页面层级应该从：

- 顶部说明卡 > 对话模块 > 输入模块

改成：

- 轻量系统状态带 > 大面积对话流 > 稳定输入控制台

## 页面结构设计

### `/chat` 新结构

建议结构如下：

1. **轻量顶部状态带**
   - 只保留一条非常轻的系统提示
   - 内容包括：
     - 来源页（如果有）
     - 系统当前身份感（例如“系统对话”）
   - 不再使用大 Hero 卡片

2. **主对话流**
   - `SystemConversation` 成为页面视觉中心
   - 高度更长、留白更舒展
   - 页面默认进入时就应该感觉“这是主界面”

3. **底部输入控制台**
   - 使用现有 `SystemChatComposer` 继续承担收发
   - 视觉进一步像主控制台：
     - 更稳的底部停靠
     - 更少“卡片感”
     - 更强“系统输入台”感

### `/system` 新结构

- 不再承担首页角色
- 保留：
  - 系统总览
  - 说明
  - 进入系统对话入口
- 页面视觉可以继续是信息型，而不是对话型

## 导航设计

### 侧边栏顺序

核心区顺序改为：

1. `/chat`
2. `/system`
3. `/today`
4. `/goals`
5. `/inbox`

理由：

- `/chat` 已经是主入口，必须在导航上反映出来
- `/system` 仍然重要，但应该变成“总览页”而不是第一入口

### 登录后默认落点

认证后默认落点改为 `/chat`。

这会影响：

- 登录后首次进入
- 任何当前把 `/system` 当首页的跳转逻辑

目标是让“打开系统就直接进入对话场”成为新的默认心智。

## 组件改造方向

### `src/app/(authenticated)/chat/page.tsx`

改造成更沉浸的页面骨架：

- 弱化当前顶部介绍块
- 让 `SystemConversation` 占据更核心的高度与宽度
- 把输入区做成更稳定的底部控制台

### `src/components/SystemConversation.tsx`

需要更偏“主场景”风格：

- 减少像普通卡片组件的感觉
- 提升作为消息主容器时的沉浸感
- 对展开最近 N 条的控件做更轻的处理，避免打断主视觉

### `src/components/SystemChatComposer.tsx`

需要从“页内输入块”进一步升级为“主输入控制台”：

- 更像系统控制台，而不是普通表单
- 视觉上要有主输入感
- 保持当前逻辑能力不变：
  - append submitting
  - request draft
  - replace last turn
  - degraded 时 focus/prefill

### `src/components/Sidebar.tsx`

需要同步两项变化：

- `/chat` 移到第一位
- `/system` 顺位后移

## 数据流与逻辑

### 保持不变的部分

以下逻辑不需要因为这次设计而重做：

- `SystemConversationProvider`
- `turnsVisible / turnsWorking`
- degraded 状态机
- localStorage 持久化
- 摘要记忆扩展点
- 偏好记忆能力

### 需要调整的部分

只有“入口与承载层”需要调整：

- 默认路由入口切换到 `/chat`
- 侧边栏第一项切到 `/chat`
- `/chat` 读取来源上下文时继续沿用当前 query 方案

## 失败与边界情况

### 1. 空会话状态

当用户首次进入 `/chat` 且没有历史时：

- 页面不能显得空
- 需要保留轻量系统引导，但不能重新退回大 Hero 介绍页

建议：

- 用轻量空状态文案直接放在对话区中
- 强调“直接对系统说你现在想推进什么”

### 2. 来源页上下文

如果 `/today`、`/profile` 带来源进入 `/chat`：

- 只做轻量提示
- 不能出现太重的顶部信息块

### 3. degraded 状态

即便系统降级理解：

- 输入和回答仍要留在同一页
- 不允许跳回 `/system` 或业务页

## 测试与验收标准

### 代码验收

- 相关文件通过 lint
- 不破坏现有对话逻辑测试
- 若新增纯函数/路由入口逻辑，补对应测试

### 手动验收

#### `/chat`

- 登录后默认进入 `/chat`
- 页面第一眼看上去是“主对话场”而不是普通内容页
- 对话流是视觉中心
- 输入区像主控制台
- 输入和回答持续留在同一页

#### 侧边栏

- 第一项为 `/chat`
- `/chat` 激活态清晰
- `/system` 退为次级页，但仍可快速进入

#### `/system`

- 仍可正常查看系统总览
- 不再有首页感压过 `/chat`

## 设计结论

这次不是单纯“美化 `/chat`”，而是把产品主入口正式切换成对话。

最终心智应该变成：

- 打开系统，先进入 `/chat`
- `/chat` 是主页面，是系统真正存在的地方
- `/system` 是辅助总览页，不再是首页

如果这个设计成立，后续用户对整个产品的第一感知会从“看系统模块”变成“直接和系统交互”。
