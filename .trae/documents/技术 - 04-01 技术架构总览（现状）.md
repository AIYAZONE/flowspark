# 技术架构总览（现状）

## 1. 一句话概括
FlowSpark 是一个基于 Next.js App Router 的 Web 应用：前端与服务端同仓（Server Actions + Route Handlers），以 Supabase 作为认证与数据后端（Auth + Postgres + Storage），并通过一组 `/api/ai/*` 路由对接 OpenAI 兼容的 AI Provider（DeepSeek/OpenAI）。PWA 与基础观测（Vercel Analytics）由 Next 配置与前端埋点提供。

## 2. 技术栈（事实清单，以代码为准）
- 运行时：Node.js 18+（见 `../../README.md`）
- 前端框架：Next.js 16.1.0（App Router）、React 19.2.3、TypeScript（见 `../../package.json`）
- 样式与 UI：Tailwind CSS v4、Radix UI、Lucide（见 `../../package.json`、`../../components.json`、`../../src/app/globals.css`）
- 数据与认证：Supabase（`@supabase/supabase-js` + `@supabase/ssr`）（见 `../../src/lib/supabase/*`）
- 动画与图表：framer-motion、recharts（见 `../../package.json`）
- PWA：`@ducanh2912/next-pwa`（见 `../../next.config.ts`）
- 观测：`@vercel/analytics`（见 `../../src/lib/analytics.ts`）

## 3. 路由与模块边界（目录地图）
### 3.1 路由分层（App Router）
- Public/入口：`src/app/page.tsx`、`src/app/layout.tsx`
- 未登录态（Auth 组）：`src/app/(auth)/*`（login/signup/forgot/reset/verify/verified）
- 登录后（Authenticated 组）：`src/app/(authenticated)/*`
  - Dashboard：`/dashboard`（写操作集中在 `src/app/(authenticated)/dashboard/actions.ts`）
  - Today：`/today`
  - Goals：`/goals`、`/goals/new`、`/goals/[id]`（写操作集中在 `src/app/(authenticated)/goals/actions.ts`）
  - Profile：`/profile`、`/profile/account`（写操作集中在 `src/app/(authenticated)/profile/actions.ts`）
- Route Handlers（API）：`src/app/api/*`（AI 相关见 `src/app/api/ai/*`）
- Auth callback：`src/app/auth/callback/route.ts`
- 并行/拦截路由（Modal）：`src/app/(authenticated)/@modal/*`（例如 `@modal/(...)goals/new`）

### 3.2 业务域（Feature Areas）
- Goals：目标与行动（actions）CRUD、筛选、详情/进度组件
- Today：今日聚焦视图、Core Action 设定、列表折叠/筛选
- Dashboard：日评分、趋势、热力图、等级/经验展示
- Gamification：经验/等级与奖励（盲盒）
- AI：目标拆解、Goal Setup、今日计划、复盘、rescue、反馈
- i18n：字典与加载（`NEXT_LOCALE` cookie）
- Analytics：前端事件上报
- PWA：workbox runtimeCaching、资源缓存策略

## 4. 访问控制（Auth）与会话刷新
- 入口：`../../middleware.ts` 调用 `../../src/lib/supabase/middleware.ts`
- 核心规则（现状）：
  - 受保护路由未登录：重定向到 `/login`
  - 受保护路由要求邮箱已验证：`user.email_confirmed_at` 为空则重定向到 `/login?error=...`
  - 已登录访问 `/login|/signup|/forgot|/reset`：重定向到 `/dashboard`
  - refresh token 异常：清理 Supabase auth cookies 并按未登录处理
  - i18n：无 `NEXT_LOCALE` 时按 `accept-language` 设默认 `zh/en`

## 5. 数据层（Supabase）概要
- Client 封装：
  - Server：`../../src/lib/supabase/server.ts`
  - Browser：`../../src/lib/supabase/client.ts`
- Schema/RLS/触发器/Storage：见 `../../supabase/*.sql` 与 `../../supabase/README.md`

## 6. AI 能力集成概要
- 形态：以 Route Handlers 提供 `/api/ai/*`（统一先 `auth.getUser()` 鉴权）
- Provider：`../../src/lib/ai/client.ts` 以 OpenAI 兼容 `/chat/completions` 调用 DeepSeek/OpenAI
- 关键配置：`AI_PROVIDER/AI_BASE_URL/AI_API_KEY/AI_MODEL/AI_TIMEOUT_MS` 与 `DEEPSEEK_*`、`OPENAI_*`

## 7. 部署与运行
- Scripts：`dev/build/start/lint`（见 `../../package.json`）
- 部署：Vercel（见 `../../vercel.json`）
- PWA：生产开启、开发禁用（见 `../../next.config.ts`）

## 8. 环境变量（仅名称与用途）
### 8.1 Supabase（必需）
- `NEXT_PUBLIC_SUPABASE_URL`：Supabase Project URL
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`：Supabase anon key（客户端可用）

### 8.2 Supabase（可选/高权限）
- `SUPABASE_SERVICE_ROLE_KEY`：用于管理级操作（例如账号删除的 admin API）；必须仅在服务端环境使用

### 8.3 AI Provider（可选，启用 AI 功能时需要）
- `AI_PROVIDER`：`deepseek | openai`（不设则按 `DEEPSEEK_API_KEY` 是否存在推断）
- `AI_BASE_URL` / `AI_API_KEY` / `AI_MODEL`：统一覆盖配置
- `DEEPSEEK_API_KEY` / `DEEPSEEK_BASE_URL` / `DEEPSEEK_MODEL`
- `OPENAI_API_KEY` / `OPENAI_BASE_URL` / `OPENAI_MODEL`
- `AI_TIMEOUT_MS`：请求超时（ms）

## 9. 已知约束与风险点
- `owner_id` 与 `user_id` 双字段兼容：RLS 与部分查询/upsert 需要同时考虑（详见《技术 - 04-04 数据模型与 RLS（现状）.md》）
- middleware 强制 email confirm：对测试/内测流程有影响，需确保 Auth 配置与邮件可达
- `SUPABASE_SERVICE_ROLE_KEY` 具备高权限：必须最小化调用面并确保不暴露到客户端
