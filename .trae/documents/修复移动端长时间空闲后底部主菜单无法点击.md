## 背景
- 底部主菜单在移动端由 [MobileNavBar.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/components/MobileNavBar.tsx#L19-L73) 渲染，点击跳转是 Next.js 的 `<Link href>`，正常情况下不会“自己失效”。
- 当前布局 [AuthenticatedLayout](file:///Users/brucewang/Documents/AIYA/goals/src/app/(authenticated)/layout.tsx#L25-L37) 是 `fixed inset-0` 的全屏容器，底部菜单 `z-50` 固定。
- 代码库内 Radix 的 Dialog/Sheet/AlertDialog overlay 也使用 `fixed inset-0 z-50`（[dialog.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/components/ui/dialog.tsx#L17-L29)、[sheet.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/components/ui/sheet.tsx#L18-L29)、[alert-dialog.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/components/ui/alert-dialog.tsx#L15-L27)）。

## 约束
- 你描述的是“长时间不操作后”才出现，符合“后台/锁屏后 JS 动画或卸载未完成导致残留遮罩层”的时间特征。
- 需要在不引入新依赖、最小改动下修复，并保证现有 Dialog/Sheet 行为不变。

## 方案
### 方案 A（优先）：透明 overlay 残留，吞掉点击
1. 复现与确认
   - 复现路径：打开任意会弹出 Sheet/Dialog 的功能（例如移动侧边栏 [MobileSidebar](file:///Users/brucewang/Documents/AIYA/goals/src/components/MobileSidebar.tsx#L68-L141) 或任一弹窗），关闭弹层后立刻锁屏/切后台 10–30 分钟，再回来点底部菜单。
   - 复现当下用元素检查器确认页面是否存在 `fixed inset-0 z-50` 的元素，且 `data-state="closed"` 或视觉上透明但仍在 DOM。
2. 修复（核心）
   - 在 3 个 overlay 组件里统一加上：`data-[state=closed]:pointer-events-none`（可选再加 `data-[state=closed]:opacity-0` 做双保险）。
   - 同步给 `DialogContent/SheetContent/AlertDialogContent` 加 `data-[state=closed]:pointer-events-none`，避免“内容残留但透明”同样吞事件。
   - 改动文件：
     - [dialog.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/components/ui/dialog.tsx)
     - [sheet.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/components/ui/sheet.tsx)
     - [alert-dialog.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/components/ui/alert-dialog.tsx)
3. 验证
   - 重复“打开弹层→关闭→立刻后台→回来点击底部菜单”的场景，确认底部菜单可点击。
   - 回归测试：正常打开弹层时 overlay 仍应拦截点击（不影响交互）。

### 方案 B（次优）：会话过期导致跳转被重定向/状态异常
1. 复现时观察 Network/地址变化：点击底部菜单是否触发到 `/login` 的重定向。
2. 如果命中：从 [supabase/middleware.ts](file:///Users/brucewang/Documents/AIYA/goals/src/lib/supabase/middleware.ts#L41-L77) 的 refresh token 清理与受保护路由重定向入手，补充前端提示或在可控时机主动刷新 session（这部分需要你接受“产品行为改变”，我会在实施前把具体改动点列出来）。

## 取舍
- 方案 A 改动最小、无需新增依赖、且能解释“偶现/长时间后才出现”的点击失效；属于 UI 层防御性修复。
- 方案 B 更偏业务/鉴权策略，可能需要 UI 提示或 session 刷新策略调整，改动面更大。

## 风险
- 给 overlay/content 增加 `data-[state=closed]:pointer-events-none` 理论上不影响正常关闭流程，但如果某些组件依赖“关闭动画期间仍拦截点击”，需要确认（通常关闭态不可交互是更合理的）。

## 建议
- 先实施方案 A（3 个 ui 组件的 class 调整）并复现验证；若仍有问题，再进入方案 B（鉴权/重定向）或 PWA/SW 的深排查。
