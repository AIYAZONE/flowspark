# 国内访问优化清单（Vercel + Supabase + Cloudflare）

## 1) 先确认瓶颈在哪里（必须做）
- 静态资源慢：HTML/CSS/JS 首次加载（TTFB、LCP）
- 动态请求慢：登录/仪表盘/目标 CRUD（API/Server Actions 到 Supabase）
- 是否每次请求都有额外鉴权开销（middleware、session refresh）

## 2) 应用侧可落地优化（不迁移架构）
### 2.1 降低每次请求的跨境往返
- 公共页面不要在 middleware 里做 Supabase 网络鉴权，只保留语言 cookie 等轻逻辑
- 只在受保护路由与认证回调路径做会话更新与鉴权

### 2.2 PWA/缓存策略（把可缓存的尽量留在本地）
- 目标：第二次打开（或弱网）依然可用、感知更快
- 建议优先缓存：Landing、Auth 页静态资源、字体、图标、基础 UI 资源
- 对需要实时数据的页面：允许“先展示上次缓存的只读视图 + 后台刷新”

## 3) 平台侧检查清单（不迁移架构）
### 3.1 Vercel
- 项目 Region：尽量选择更靠近大陆用户的区域（例如亚洲区域），降低 TTFB
- 静态资源缓存：确保 `_next/static/*` 有长期缓存头（Vercel 默认通常已处理）
- 观测：开启 Web Vitals 采样与关键页面 LCP/TTFB 分析

### 3.2 Supabase
- 项目 Region：优先选择靠近目标用户的区域（大陆用户通常选择新加坡/东京/香港更友好）
- Auth 与数据库：确认登录/仪表盘关键请求的 P95 延迟
- 存储桶资源：若大量使用 Storage 图片/头像，确认其对大陆可用性（必要时改用可替代对象存储/CDN）

### 3.3 Cloudflare（仅 DNS/解析与基础缓存场景）
- 仅 DNS 解析（灰云）：通常更简单、更接近 Vercel 官方推荐的直连方式
- 代理（橙云）：可能引入额外链路复杂性；如果启用，需要明确缓存规则与回源策略，避免对动态鉴权页面缓存
- HSTS/HTTPS 重定向：确保一致，不要产生多跳 301/302

## 4) 结构性方案（真正面向大陆体验，但属于“迁移/重构”范畴）
- 前端部署迁移到更贴近大陆的边缘网络/云厂商
- 数据与鉴权迁移到香港/新加坡自托管或替代方案
- 引入国内可用 CDN/加速网络与合规流程（需要运营侧配合）
