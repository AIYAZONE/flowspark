## 背景
- 底部主菜单是 [MobileNavBar.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/components/MobileNavBar.tsx#L45-L73) 的 `position: fixed` + `z-50`，点击靠 Next.js `<Link />`。
- 目前已对 Dialog/Sheet/AlertDialog 的关闭态加了 `pointer-events-none`，但你反馈问题仍存在，说明“遮罩处于 closed 仍吞点击”不是唯一原因。
- 代码库还存在两个值得重点怀疑的“会遮挡/导致点击异常”的点：
  - 根级全屏 loading： [app/loading.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/app/loading.tsx#L4-L13)（`fixed inset-0 ... z-50`）。
  - Radix Select 的 Portal 内容： [select.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/components/ui/select.tsx#L121-L151)（`Portal` + `z-50`，且未做 closed 态禁用 pointer-events）。
- 另外，当前 authenticated 页面结构是「外层固定全屏 + 内层 main 滚动 + 底部 nav fixed」。iOS/Safari/PWA 在后台恢复后，`fixed` 元素在复杂滚动/合成层场景下出现“偶发不可点”是高频坑（不一定有遮罩残留）。

## 约束
- 不新增依赖。
- 变更尽量小、以 diff 为主。
- 修复要可验证：至少能定位“到底是没收到点击”还是“收到点击但路由不跳”。

## 方案
### 方案 A（优先，结构性根治）：去掉底部菜单的 fixed
- 把底部菜单从 `fixed bottom-0 ...` 改为“正常布局中的底部栏”（`shrink-0` / `sticky bottom-0` 二选一），利用 [AuthenticatedLayout](file:///Users/brucewang/Documents/AIYA/goals/src/app/(authenticated)/layout.tsx#L25-L35) 的 `flex-col` 结构，让 nav 永远在 main 下方，不再依赖 fixed。
- 同步调整 main 的 `pb-24`（原本是给 fixed nav 让位），改为更合理的 padding（或只保留 safe-area padding）。
- 目标效果：即使 iOS 合成层/滚动 bug 触发，底部栏作为普通布局元素也更稳定；同时也更不容易被“同级 z-index 的 Portal”抢占事件。

### 方案 B（增强稳定性）：后台恢复时主动收敛潜在弹层/Portal
- 新增一个轻量 client 组件挂在 authenticated layout（例如 `AppResumeGuards`）：
  - 监听 `visibilitychange/pageshow`，当页面从后台恢复为可见时：
    - 如果存在 Radix Portal 且内部有 `data-state="open"` 的节点，则向 document 派发一次 `Escape` 键事件（让 Select/Dialog/Sheet 统一收敛）。
    - 触发一次底部栏的 remount（通过 key/state），强制重建事件绑定与渲染层。
- 目标效果：解决“后台恢复后某个 Portal/弹层仍处于 open（或视觉异常）导致吞点击/层级异常”的场景。

### 方案 C（补全你已同意的指针事件修复范围）：SelectContent 关闭态禁用 pointer-events
- 在 [select.tsx](file:///Users/brucewang/Documents/AIYA/goals/src/components/ui/select.tsx#L125-L149) 的 `SelectPrimitive.Content` class 增加 `data-[state=closed]:pointer-events-none`，与 Dialog/Sheet 的防御逻辑一致。

### 方案 D（可选，仅用于快速定位）：底部栏点击诊断开关
- 给 MobileNavBar 加一个仅在 `?debugNav=1` 时生效的捕获逻辑：
  - `onPointerDownCapture` 里用 `document.elementFromPoint(x,y)` 打印“实际命中的最上层元素”。
  - `onClick` 里打印“是否触发、pathname 是否变化”。
- 目标效果：一旦再复现，可以明确到底是“点不到（事件被遮挡）”还是“点到了但路由不走/被卡住”。

## 取舍
- A 改动最少但属于布局策略调整，属于更像“根治 iOS fixed 偶发不可点”的方向。
- B 会在后台恢复时主动关掉残留弹层，可能改变极少数用户“切后台再回来还想继续在弹层里操作”的体验，但对稳定性收益很大。
- C 是纯防御补全，几乎无风险。
- D 不改变业务，仅用于定位；可默认关闭。

## 风险
- A 需要小心 safe-area 与内容底部留白（避免内容被底部栏挤压或出现多余空白）。
- B 的 Escape 收敛必须加条件（只在检测到确实有 `data-state=open` 的 portal 时触发），避免误伤。

## 建议
- 直接实施 A + C + B（B 带条件），并保留 D 作为开关便于线上/真机复现定位。
