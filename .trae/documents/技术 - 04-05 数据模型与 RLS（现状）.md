# 数据模型与 RLS（现状）

## 1. 数据源与执行顺序
数据库初始化脚本位于 `../../supabase/`（见 `../../supabase/README.md`），按顺序执行：
- `00_init_tables.sql`：核心表结构
- `01_rls_policies.sql`：核心表 RLS
- `02_storage_setup.sql`：Storage bucket 与策略
- `03_functions_and_triggers.sql`：新用户触发器 + `updated_at` 自动维护
- `04_gamification.sql`、`05_gamification_rls.sql`：游戏化最小落库与 RLS
- `06_ai_feedback.sql`：AI 反馈字段扩展

## 2. 核心表（摘要）
### 2.1 `user_profiles`
- 主键：`id`（uuid，引用 `auth.users(id)`）
- 典型字段：`name`、`avatar_url`、`timezone`、`locale`、`created_at/updated_at`
- 扩展字段：
  - `xp`、`level`（见 `04_gamification.sql`）
  - `ai_recent_events`（jsonb，见 `06_ai_feedback.sql`）

### 2.2 `goals`
- 主键：`id`
- 归属字段：`owner_id` 与 `user_id` 并存（兼容策略，详见第 4 节）
- 关键字段：`title/description/start_date/end_date/success_criteria/stop_criteria/status/priority/category`
- 关系：`actions.goal_id -> goals.id`

### 2.3 `actions`
- 主键：`id`
- 归属字段：`owner_id` 与 `user_id` 并存
- 关系：`goal_id -> goals.id`（删除目标会级联删除 actions）
- 约束：`type` 仅允许 `core|maintenance|learning|review|rest`（见 `00_init_tables.sql`）

### 2.4 `daily_scores`
- 主键：`id`
- 归属字段：`owner_id` 与 `user_id` 并存
- 业务唯一性：`UNIQUE (owner_id, score_date)`（用于 upsert）
- 代码侧 upsert：优先 `onConflict: 'owner_id, score_date'`，失败时 fallback 到 `user_id, score_date`（见 `../../src/app/(authenticated)/dashboard/actions.ts`）

### 2.5 `xp_logs`（游戏化）
- 主键：`id`
- 归属字段：`user_id`（此表不使用 owner_id 兼容）
- 关系：`action_id -> actions.id`（action 删除时 `SET NULL`）
- 索引：`(user_id, created_at DESC)`、`(action_id)`（见 `04_gamification.sql`）

## 3. RLS（Row Level Security）策略摘要
### 3.1 核心表（goals/actions/daily_scores/user_profiles）
策略目标：用户只能访问自己的数据（基于 `auth.uid()`）。
- `user_profiles`：`auth.uid() = id`
- `goals/actions/daily_scores`：`auth.uid() = user_id OR auth.uid() = owner_id`
对应脚本：`01_rls_policies.sql`

### 3.2 游戏化表（xp_logs）
- `xp_logs`：`auth.uid() = user_id`
对应脚本：`05_gamification_rls.sql`

### 3.3 Storage（avatars bucket）
- Public Read：任何人可读 `bucket_id='avatars'`
- 写入/更新/删除：仅认证用户且 `owner = auth.uid()`
对应脚本：`02_storage_setup.sql`

## 4. `owner_id` vs `user_id` 兼容策略（现状）
### 4.1 背景
Schema 与代码存在历史演进：部分逻辑以 `owner_id` 作为主归属字段，但为兼容旧表/旧约束，同时保留 `user_id` 并在错误时 fallback。

### 4.2 代码侧体现（示例）
- Goals/Actions 写操作：优先写 `user_id + owner_id`；遇到“列不存在/约束不匹配”等错误，尝试仅写 `user_id` 或仅写 `owner_id`（见 `../../src/app/(authenticated)/goals/actions.ts`）
- Daily Scores：优先以 `(owner_id, score_date)` 唯一约束 upsert；失败再以 `(user_id, score_date)` upsert（见 `../../src/app/(authenticated)/dashboard/actions.ts`）

### 4.3 维护建议（不改变现有接口前提下）
- 新表若涉及“归属”，明确选用单一字段（推荐 `user_id`），避免继续扩大兼容面。
- 若要彻底消除双字段：需要一次性完成数据回填、索引/唯一约束重建、RLS 简化，并同步移除 fallback 分支（属于重构议题，不在本文档范围内）。

## 5. 触发器与自动化
对应脚本：`03_functions_and_triggers.sql`
- 新用户：`auth.users` 新增记录时，自动插入 `public.user_profiles`
- 时间戳：对 `user_profiles/goals/actions/daily_scores` 的更新自动刷新 `updated_at`
