# ADR-0001 技术栈与数据策略

## 背景
- 需要快速交付 MVP，保障可维护性与安全；支持后续扩展

## 问题
- 全栈一体化与服务端行为如何组织？数据权限如何保证？

## 备选方案
- Next.js + Server Actions + Supabase（Postgres+Auth+Storage）
- 传统 REST/BFF + 外部 Auth 服务

## 权衡
- 选择 Server Actions：类型安全、减少 API 层、UI 与写库更紧密
- 选择 Supabase：低运维、RLS 原生、生态成熟；避免自建复杂度

## 决策
- 采用 Next.js App Router + Server Actions；Supabase 作为 BaaS
- i18n 自研轻量方案；Zustand 管理 UI 局部状态

## 后果
- 复杂业务逻辑需在 Server Actions 中保持良好分层与校验
- 数据迁移脚本与 RLS 策略需同步维护与验证

## 后续行动
- 编写 API 与数据字典、测试与质量策略、运维与监控文档并联动
